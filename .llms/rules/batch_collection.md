# Rule: Corpus inventory must start from committed layers, not scratch

**Problem this rule prevents**

Agents naturally enter through `.data/samples/` because that's where the bytes live, but that directory has zero status metadata. It mixes:
- accepted output (`datasets/` built)
- staging experiments
- already-removed recipes
- orphan exploratory pulls
all as identical `<id>/*.bin`. Because `.data/` is gitignored, `git status` / `git log` provide no provenance. Inventorying from `.data/` leads to false positives ("unlisted" dataset) and confuses OOD families as suspicious.

Secondary trap: checking "is it in corpus?" by matching `source_sample` / `sample_path` strings from manifests or `.data/index/*/samples.jsonl`. Those paths break on rename/rebuild and cause false negatives.

**Authoritative sources ŌĆö always start here**

- Accepted corpus: `datasets/<dataset_id>/manifest.toml` ŌĆö only these directories count
- Drafts: `staging/<dataset_id>/` (ignored except `staging/README.md`)
- Registry: `attempts/dataset_status.tsv` + `attempts/README.md` ŌĆö statuses `accepted | rejected | blocked | deferred | transient_failure | needs_tooling | superseded`
- Audit / sizes: `reports/accepted_recipe_audit.tsv` + `.md` (generated by `tools/audit_acceptance.py`)
- Logs for evidence: `.data/logs/<id>/download.latest.log`, `build.latest.log`, `verify.latest.log` ŌĆö only after grounding on registry

**Execution steps for any batch_collection / audit / family-coverage task**

1. List committed recipes: `ls datasets/` or parse `reports/accepted_recipe_audit.tsv`
2. Parse `attempts/dataset_status.tsv` for blocked/rejected/deferred/transient_failure/needs_tooling/superseded
3. Cross-reference by **stable `dataset_id`** and registry fields `replacement_id`, `active_path` ŌĆö NOT by `source_sample`, `sample_path`, or `.data/index/...` volatile paths
4. Only after that, optionally inspect `.data/samples/` / `.data/index/` for byte-level debugging of a specific candidate you already grounded

**Hard rule**

- NEVER call `ls .data/samples/` as proof of corpus membership
- NEVER enumerate `.data/samples/` to judge family coverage or quality
- NEVER use `source_sample` / `sample_path` exact match as "is it listed?" check
- ALWAYS cite `datasets/`, `attempts/dataset_status.tsv`, `reports/accepted_recipe_audit.tsv` as evidence

**Signpost**

See `.data/README.md` (tracked via `!.data/README.md`) for the same warning at the exact layer where confusion happens. See `AGENTS.md` authoritative vs ephemeral layers section for repo-wide rule.

**Cross-ref hardening**

If you maintain audit tooling under `tools/`, key on stable IDs (`dataset_id`, registry `family`) instead of manifest `source_sample` paths. Volatile paths cause under-reporting of wired families as "unlisted".

This file is committed so all future agents working this repo share the same guardrail, not just one agent's persistent memory.
