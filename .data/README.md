# Ephemeral Scratch — NOT the Corpus

**A directory under `.data/` existing does NOT mean the dataset is accepted, usable, or even valid.**

`.data/` is local, gitignored scratch that mixes:
- accepted recipe output (`datasets/` built successfully)
- staging experiments (`staging/` drafts)
- already-removed-recipe remnants
- orphan exploratory pulls and partial downloads
all as identical `<id>/*.bin` under `samples/`, `filtered/`, `downloads/`, etc.

Because `.data/` is gitignored, `git status` / `git log` give **no signal** that it is ephemeral. It is the natural place to start looking for bytes, but it is exactly the layer with the least provenance.

## Authoritative layers — check these first

1. **Accepted corpus:** `datasets/<dataset_id>/manifest.toml`
   - Only directories under `datasets/` are accepted recipes.
   - Each must have passed user-run `download.sh` + local `build.sh` + `verify.sh`.

2. **Draft area:** `staging/<dataset_id>/`
   - By design git-ignored except `staging/README.md`.
   - Do NOT promote to `datasets/` until the three steps above pass.

3. **Attempt registry (negative results):** `attempts/dataset_status.tsv`
   - Machine-readable: `dataset_id | status | active_path | evidence_path | replacement_id | reason | retry_condition`
   - Statuses: `accepted`, `rejected`, `blocked`, `deferred`, `transient_failure`, `needs_tooling`, `superseded`.
   - Check this **before** treating any historical report, runbook, or `attempts/*.md` as a current recommendation.
   - See `attempts/README.md` for file naming convention.

4. **Accepted-recipe audit (sizes):** `reports/accepted_recipe_audit.tsv` and `.md`
   - Generated by `tools/audit_acceptance.py` from `datasets/*/manifest.toml` + `.data/index/*/samples.jsonl`.
   - Primary-values, sample-bytes, median sample size, has_index — the real acceptance floor view.

5. **Protocol & case law:** `collection_protocol.md`, `reports/protocol_case_law.md`, `reports/family_homogeneity_policy.md`, `datasets/README.md`, `staging/README.md`.

## How to judge what is in the corpus — DO NOT use .data/ alone

- To list accepted datasets: `ls datasets/` or `cat reports/accepted_recipe_audit.tsv`.
- To check if an ID is accepted/blocked/rejected: `grep <id> attempts/dataset_status.tsv`.
- To check build state: `ls .data/index/<id>/samples.jsonl` only proves *some* build ran; you still need `attempts/dataset_status.tsv` status and `datasets/<id>/` existence.
- To check natural-record boundaries / scanner output: `python3 tools/report_dataset_state.py datasets/<id>`.

## Do NOT

- Do NOT enumerate `.data/samples/` and treat each subdir as "a dataset".
- Do NOT infer families or quality from file counts / sizes in `.data/samples/`.
- Do NOT match on `manifest.toml` `source_sample` paths or `sample_path` strings from `.data/index/` — those break when recipes are renamed/rebuilt. Key on stable `dataset_id` and registry `family`/`replacement_id` instead.

## Lifecycle of .data/ subdirectories

- `downloads/` — raw upstream zips/tifs/etc per `download.sh`
- `extracted/` — unpacked
- `filtered/` — bounded subsets
- `samples/` — final `.bin` primary samples (decodelt typed values, not opaque file bytes)
- `index/` — `samples.jsonl` per dataset: `dataset_id, series_id, sample_path, numeric_kind, bit_width, value_count, sample_size_bytes`
- `logs/` — `download.latest.log`, `build.latest.log`, `verify.latest.log`
- `batches/` — temporary batch launch summaries (outside repo by convention)

All can be safely deleted and rebuilt via `download.sh` + `build.sh`. Nothing under `.data/` except this README is committed.

## If you are an agent newly entering this repo

1. Read `AGENTS.md` and `collection_protocol.md`.
2. Read `datasets/README.md` and `staging/README.md`.
3. Read `attempts/README.md`.
4. Use `reports/accepted_recipe_audit.tsv` and `attempts/dataset_status.tsv` as ground truth.
5. Only then look at `.data/` for byte-level debugging, not for inventory.

This file is force-tracked via `!.data/README.md` in `.gitignore` precisely so agents hitting the scratch layer get redirected to the authoritative layers.
